# Agents and Generated Content Policy (Digital Media Equipment Tracker)

AI agents must read and follow this file before making any changes. Failure to follow these rules invalidates the generated output.

IMPORTANT: Agents MUST read the entire contents of this file and explicitly follow every rule, checklist item, and requirement it lists. If the file instructs an agent to perform specific steps (for example: include metadata in PRs, add tests, run linting, or avoid committing secrets), the agent is responsible for completing those steps or clearly documenting why a step couldn't be completed. Any deviation must be recorded in the PR description along with the reason. Failure to comply may result in generated changes being reverted and the contribution being rejected.

## Purpose

This project accepts contributions that may include content generated by automation or AI-assisted tools ("agents"). This document defines the mandatory rules collaborators must follow whenever any content (code, docs, tests, config, text) is produced by an agent so that generated material is high-quality, safe, and auditable.

## Scope

Applies to:

- Any file added, modified, or removed where an automated agent produced or suggested content
- All collaborators who review, open, or merge pull requests that include generated content

## Agents used in this repository

- CI automation: GitHub Actions (`.github/workflows/ci.yml`)
- Local tooling: HTML, CSS, JavaScript (vanilla or framework-based)
- Backend/database: Supabase (database, auth, real-time subscriptions)
- Any AI or code generation tools used by contributors (e.g., Claude, Copilot, Cursor)

## Mandatory rules for generated content

1. **Human review required**
   - Every change produced or suggested by an automated agent must be included in a pull request and reviewed by at least one human reviewer before merge (unless merged by an approved bypass actor).
   - PR description must include a short summary of what the agent changed, why the change was made, and which agent/tool/version produced it.

2. **Tests and checks (explicit expectations)**
   - Generated code must include tests that exercise the new functionality where feasible.
   - For UI or user-flow changes, include manual testing steps or automated end-to-end tests that cover the main flow.
   - Tests must be runnable locally; target minimum local test commands:
     - Open `index.html` in browser (or run local dev server)
     - Verify Supabase connectivity with test credentials
     - Walk through checkout/check-in flow manually
   - If immediate full coverage is impractical, include focused tests covering critical paths and add a short rationale in the PR notes explaining remaining gaps and a plan to add more tests.

3. **Build and lint (concrete checks)**
   - All generated code must be syntactically valid and run without console errors in modern browsers (Chrome, Firefox, Edge).
   - All JavaScript must pass linting if a linter is configured (e.g., ESLint).
   - CSS should be validated and avoid deprecated properties.

4. **No secrets (hard prohibition)**
   - Never commit secrets, API keys, credentials, or private certificates into the repository.
   - Use environment variables or a `.env` file excluded via `.gitignore`. Include a `.env.example` (without real values) so developers know required variables.
   - Supabase anon keys (public, row-level-security-protected) may be stored in config files, but **service role keys must never be committed**.
   - If a secret is accidentally added, immediately remove it, rotate the secret, and document the incident in the PR.

5. **Documentation (what to include)**
   - Update any user-facing or developer docs affected by generated changes (README, inline comments, or doc blocks).
   - PR description must include: tool name and version, prompt used, files added/changed, tests added, and a short instruction for running/testing locally.

6. **Commit hygiene (format and messages)**
   - Keep generated changes in small, focused commits where possible. Each commit message must be descriptive.
   - When an agent updates multiple unrelated areas, split into multiple PRs or commits and explain why they were grouped.
   - Never add AI agents (e.g., Claude, Copilot, or any automated tool) as co-authors in commit messages or pull requests. Do not include `Co-Authored-By` or `co-authored-by` trailers referencing any AI agent. All authorship must belong to human contributors only.

7. **Licensing and attribution (specific steps)**
   - Verify that any generated content or third-party snippets comply with the repository license.
   - If external code, libraries, or assets are used that require attribution, include a note in the PR and (if required) a LICENSE or ATTRIBUTION file.

8. **Security and dependencies (review steps)**
   - When adding or updating dependencies (npm packages, CDN scripts, Supabase client versions), list them in the PR.
   - Prefer pinning versions where reproducibility matters; avoid wildly permissive ranges for security-sensitive dependencies.
   - Validate that any third-party scripts loaded via CDN are from trusted sources and use integrity hashes when possible.

9. **Reproducible generation (metadata)**
   - Every PR that includes generated content must include metadata in the PR description: tool name, tool version, generation prompt (or a link to it), and the generation timestamp.
   - Store the metadata in a reproducible format (e.g., a short YAML snippet in the PR) to allow auditing later.

10. **Rollback plan (concrete guidance)**
    - For changes that alter behavior or data flows, include a short rollback plan in the PR: which commits to revert and how to restore previous functionality.
    - For generated code that introduces new features, include a feature flag or toggle recommendation when feasible.

11. **Input validation**
    - Validate all user inputs strictly (student names, equipment names, quantities, timestamps).
    - Use type checks, length limits, and sanitization to prevent injection or malformed data.

12. **Supabase / API key handling**
    - Never hardcode service role keys or admin credentials.
    - Store connection details in environment variables only.
    - Ensure keys are never exposed in client-side code beyond the public anon key.
    - Use Supabase Row Level Security (RLS) policies to protect data at the database level.

13. **Tests for generated code**
    - Whenever an agent generates code (new files or modifications that add functionality), the agent must also add corresponding tests or manual testing instructions that exercise the generated code.
    - The agent should include a short note in the PR description explaining what tests were added and what they cover, plus any rationale for omitted tests if coverage cannot be achieved immediately.

14. **CI and checks (specific requirements)**
    - All generated-code PRs must include a passing CI run that at minimum executes linting and any configured automated tests.
    - Manual testing steps should be documented in the PR for UI-heavy changes.

15. **Code style and formatting**
    - Use consistent formatting (Prettier or equivalent) before committing generated code.
    - Follow the project's established naming conventions and file organization.

16. **Performance and resource usage**
    - Avoid adding code that performs blocking or expensive operations on the main thread.
    - Minimize unnecessary Supabase API calls; use batching and caching where appropriate.
    - Optimize for the classroom environment (potentially slower networks, shared devices).

17. **Accessibility**
    - All generated UI must be keyboard-navigable and screen-reader friendly.
    - Use semantic HTML elements, ARIA labels where needed, and sufficient color contrast.
    - Forms must have proper labels and error states.

18. **Documentation template for PRs with generated content**
    - A suggested PR description template:
      ```
      - Tool: <name> <version>
      - Prompt: <short prompt or link>
      - Files changed: <list>
      - Tests added: <paths and descriptions>
      - How to test locally: <steps>
      - Rationale for omitted tests (if any): <notes>
      ```

19. **Auditing and revertability**
    - Keep generated changes small and auditable. If a generated change is large, include a summary and consider splitting into feature branches.
    - Maintain a clear revert plan and use tags/branch names that indicate generated content (e.g., `generated/<feature>`).

20. **README updates when setup steps change**
    - Whenever a code change affects commands that a developer must run (e.g., new dependencies, new environment variables, changed startup commands, new scripts), the agent must update `README.md` in the same change to reflect the new commands.
    - A developer following the README must be able to set up and run the project successfully without needing to read source code.

21. **README and documentation updates for large changes**
    - When an agent or contributor makes a large change (major feature additions, refactors, API/schema changes), update the repository README and any affected documentation as part of the same PR.
    - If the change is substantial, add a short "Notable changes" or "Changelog" subsection summarizing what changed and any developer actions required.

## Project-specific rules

### Stack constraints

- **Frontend**: HTML, CSS, JavaScript (vanilla JS or a lightweight framework like React/Vue if the team chooses)
- **Backend/Database**: Supabase (PostgreSQL, Auth, Realtime)
- **Hosting**: Static hosting (GitHub Pages, Netlify, Vercel, or similar)
- **Development**: VS Code as the primary editor

### Folder structure

```
project-root/
├── index.html              # Main entry point
├── css/                    # Stylesheets
│   └── styles.css
├── js/                     # JavaScript modules
│   ├── app.js              # Main application logic
│   ├── supabase.js         # Supabase client initialization
│   ├── students.js         # Student management logic
│   ├── equipment.js        # Equipment/inventory logic
│   └── checkouts.js        # Checkout/check-in logic
├── assets/                 # Images, icons, fonts
├── tests/                  # Test files (if using a test framework)
├── .env.example            # Environment variable template
├── .gitignore
├── agents.md               # This file
├── prd.md                  # Product requirements document
└── README.md               # Setup and usage instructions
```

### Supabase rules

- Use a centralized Supabase client initialized in `js/supabase.js`
- Do not hardcode Supabase URLs or keys in any file other than `.env`
- Enable and configure Row Level Security (RLS) on all tables
- Use Supabase Realtime for live updates to checkout status when feasible

### Branching rules

- Do not push directly to `main`
- Always use a feature branch
- Open a pull request for all changes
- Non-admin users require approval to merge
- Admins may bypass reviews but must ensure CI passes

### Testing expectations

- Manual testing steps must be documented for all UI features
- Automated tests should cover core logic (checkout/check-in calculations, input validation)
- Do not remove tests unless replacing with better coverage

### Performance and safety

- Avoid unnecessary Supabase API calls
- Validate all inputs on both client and database level (RLS + constraints)
- Handle errors explicitly and display user-friendly error messages
- Design for a classroom environment: fast load times, clear UI, minimal training needed

## PR review checklist

- [ ] No secrets or credentials included
- [ ] Code runs without errors in browser console
- [ ] Linting passes (if configured)
- [ ] Manual testing steps documented and verified
- [ ] Automated tests added or updated (if applicable)
- [ ] Documentation updated
- [ ] Generation details included in PR
- [ ] Code reviewed by a human or approved bypass actor
- [ ] Accessibility checked (keyboard nav, labels, contrast)

## Best practices for reviewers

- Review changes in small chunks
- Verify test coverage and manual testing steps
- Check for secrets or large unintended files
- Request prompts or context if unclear
- Test the UI in at least two browsers

## Enforcement

Maintainers may block or revert PRs that do not follow this policy. Branch protection rules should enforce CI checks and required reviews.

## Location and application

This file is the single source of truth for generated content policy in this repository. Changes to this file must follow the same review and testing rules as code changes.
